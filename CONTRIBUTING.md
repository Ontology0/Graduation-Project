# Contributing Guide

## 브랜치 전략

```
main ← 최종 제출 / 배포용
  └── dev ← 일상 개발 / PR 통합용
        ├── feat/data/설명
        ├── feat/dpo/설명
        ├── feat/rag/설명
        ├── feat/eval/설명
        ├── docs/설명
        ├── chore/설명
        └── fix/설명
```


### 작업 순서 (일상)
```
1. dev 브랜치에서 새 브랜치 생성
2. 작업 후 dev로 PR
3. 팀원 리뷰 후 merge
```

### dev → main 반영 시점

`main`은 **안정된 스냅샷**만 두고, 일상 작업은 모두 `dev`에서 진행한다. 아래 중 하나에 해당할 때 `dev`를 `main`으로 올린다.

| 시점 | 예시 |
|---|---|
| 마일스톤 완료 | 데이터 전처리 파이프라인 완료, RAG 1차 동작 확인 등 |
| 실험·제출 전 | 과제 제출, 데모, 논문용 결과 고정이 필요할 때 |
| 팀 합의 | “이번 주 범위는 여기까지”처럼 반영 범위를 맞춘 뒤 |

**올리기 전 체크**
- `dev`에 머지된 PR이 리뷰를 거쳤는지 확인
- 로컬에서 핵심 스크립트·테스트가 통과하는지 확인 (가능한 범위에서)
- API 키·대용량 데이터·`.env` 등이 포함되지 않았는지 확인

**반영 방법**
```
1. GitHub에서 dev → main PR 생성
2. 변경 요약·마일스톤 내용을 PR 본문에 간단히 기록
3. 팀원 1명 이상 리뷰 후 merge (본인 단독 merge 지양)
4. merge 후 로컬: git checkout main && git pull
```

`feat/*`, `docs/*` 등 작업 브랜치는 **main으로 직접 PR하지 않는다.** 항상 `dev`를 거친 뒤, 위 기준에 맞을 때만 `dev` → `main`으로 올린다.

## 커밋 메시지 규칙

형식: `태그: 내용(#이슈번호)`

| 태그 | 언제 |
|---|---|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `refactor` | 코드 구조 변경 |
| `chore` | 설정, 패키지, 파일구조 변경 등 기타 |

### 예시
```
feat: ClashEval 전처리 스크립트 추가(#2)
fix: RAG 파이프라인 임포트 오류 수정(#3)
docs: README 실험 설계 업데이트(#13)
chore: doc/ 하위 폴더 구조 추가(#99)
```

## 주의사항

- `main`에 직접 push 금지 (`dev` → `main`도 PR로만 반영)
- 작업 브랜치 → `main` 직접 PR 금지 (`dev` 경유)
-  issue를 먼저 생성 후 작업하는 것을 권장
- `.DS_Store`, `.env` 커밋 금지 (.gitignore에 기록해서 깃허브로 안 넘어가게 관리)
- PR은 본인이 merge하지 말고 팀원 리뷰 후 merge
- PR 올리기 전에 로컬에서 먼저 테스트
