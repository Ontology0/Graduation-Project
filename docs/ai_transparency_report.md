# AI 투명성 리포트

> 본 문서는 프로젝트 개발 과정에서 AI 도구를 활용한 내역을 투명하게 기록합니다.

> ⚙️ **메타 투명성:** 본 보고서 자체도 Claude Code(claude-sonnet-4-6)의 보조로 작성되었습니다. 팀이 작성 기준과 방향을 먼저 제시하고, AI가 초안을 작성하면 팀이 검토·수정하는 방식으로 운영되었습니다. 이 사실 자체가 우리 팀의 AI 활용 원칙(팀 주도 → AI 실행 → 팀 검증)을 보여주는 예시입니다.

---

## 0. 본 문서의 작성 기준

본 보고서는 아래 5가지 기준을 토대로 작성되었습니다.

| 기준 | 핵심 질문 |
|------|-----------|
| **사용 투명성** | AI를 어디에 썼는지 솔직하게 기록했는가? |
| **활용 적절성** | AI가 필요한 곳에 적절하게 쓰였는가? |
| **이해도** | AI가 만든 결과를 팀이 설명할 수 있는가? |
| **검증성** | AI 결과를 실행·비교·테스트·근거 확인으로 검증했는가? |
| **책임성** | 오류, 한계, 보안, 개인정보 위험을 인식했는가? |

**핵심 원칙:** 팀이 연구 방향과 판단 기준을 먼저 설정하고, AI는 구현 보조와 반복 작업 자동화에 활용하였습니다. 연구 문제 정의, 실험 설계 방향, 결과 해석, 모델 선택, 평가 기준은 팀이 직접 결정하였습니다.

## 1. Human-in-the-Loop 체크포인트

AI 출력이 연구에 반영되기 전에 팀이 반드시 개입하는 지점을 명시합니다.

| 단계 | AI 수행 | 팀 개입 (필수) |
|------|---------|---------------|
| 선행 연구 분석 | 논문 요약, 방법론 정리 | 연구 방향 채택 여부 결정, 원문 수식·수치 직접 대조 |
| 코드 구현 | 모듈 초안 생성 | 코드 리뷰, 실행 테스트, 로직 수정 |
| 실험 설계 | 실험 코드 작성 보조 | RQ 정합성 판단, 재설계 결정, 결과 해석 |
| 문서 작성 | 초안 생성 | 내용 검토, 표현 수정, 공개 범위 결정 |
| 데모·배포 | 코드 구현 | 시나리오 설계, 보안 정책 결정, 최종 확인 |
| **이 보고서** | 초안 작성 | **팀이 평가 기준 제시 → AI 작성 → 팀 검토·수정** |

| `outputs/`, `scripts/`, `tests/` 추가 | 디렉토리 생성, CLI 스크립트 작성 | 결과물 위치 및 엔트리포인트 구조 설계 |

### 2.3 RAG 파이프라인 구현

| 항목 | 내용 |
|------|------|
| **사용 도구** | Claude Code (claude-sonnet-4-6) |
| **활용 내용** | 팀이 설계한 모듈 구조와 인터페이스 명세를 바탕으로 각 모듈 코드 초안 생성 |
| **팀의 판단** | **임베딩 모델(all-MiniLM-L6-v2), 벡터 스토어(FAISS), 로컬 테스트 모델(phi-2), 최종 타겟 모델(Llama 3.1-8B) 선택은 모두 팀의 연구적 판단**. Conflict-Aware 프롬프트 내용은 팀이 직접 설계 후 AI가 코드로 구현 |

**팀이 검증한 항목:**
- 각 모듈의 입출력 인터페이스가 설계 명세와 일치하는지 코드 리뷰
- phi-2 로컬 실행으로 전체 파이프라인(Load→Chunk→Embed→Retrieve→Generate) 동작 확인
- `outputs/runs/` 결과물 JSON 구조 및 필드명 검토

### 2.4 HuggingFace Spaces 인터랙티브 데모 구축

### 2.5 파일럿 실험 (API 모델 기반)

| 항목 | 내용 |
|------|------|
| **사용 도구** | OpenAI gpt-4o-mini, Anthropic claude-haiku (실험 대상 모델), Claude Code (실험 코드 작성 보조) |
| **활용 내용** | 실험 실행 코드 초안 생성, 결과 집계 스크립트 작성 보조 |
| **팀의 판단** | **실험 설계 전체 팀 주도**: exp1→exp4 점진적 설계, A/B/C 타입 분해 아이디어 모두 팀 발안. **exp2에서 "거짓 거부"와 "최신 수용"이 RQ와 방향이 반대임을 팀이 발견** → exp3 temporal 재설계 결정. 강한 모델 천장 효과를 "상한선 확정"으로 재해석하고, **Llama 3.1-8B를 본 연구의 진짜 측정 대상으로 확정**한 것도 팀의 연구적 판단 |

**팀이 수행한 검증:**
- 각 케이스별 gold answer와 모델 출력 직접 대조 채점
- 천장 효과 발생 원인(충돌 신호가 명시적) 분석 및 데이터 설계 교훈 도출
- 결과 수치(75%→100%, 50%→83%)의 연구적 의미 팀이 직접 해석

## 3. 인간 주도 핵심 연구 영역

다음 영역은 AI 보조 없이 팀이 직접 수행하였습니다.

| 영역 | 팀의 기여 내용 |
|------|--------------|
| **연구 주제 설정** | PA-RAG의 knowledge conflict 누락을 팀이 식별, 명시적 정렬 축으로 추가하는 연구 방향 수립 |
| **핵심 가설** | conflict-aware prompting 효과를 DPO+LoRA로 내재화하면 도메인 일반화 성능이 향상된다는 가설 수립 |
| **실험 설계·재설계** | exp1→exp4 점진적 설계, exp2에서 RQ 불일치 발견 후 temporal 시나리오로 pivot |
| **결과 해석** | 천장 효과를 "상한선 확정"으로 재해석, Llama-8B가 진짜 측정 구간임을 판단 |
| **모델 선택** | phi-2(로컬 smoke test), Llama 3.1-8B(타겟), API 모델(상한선 측정) 선택 근거 결정 |
| **평가 기준** | conflict resolution 정답 기준 설계, substring 매칭 한계 인식 및 대안 지표 검토 |

## 4. AI 시스템 한계 인식 및 대응

| 한계 | 팀의 인식 및 대응 |
|------|-----------------|
| **Hallucination** | AI 생성 코드·요약·수치를 무비판적으로 수용하지 않고 원문·실행 결과와 대조 검증 |
| **파일럿 수치의 한계** | gpt-4o-mini·claude-haiku 기반 수치는 API 모델 상한선이며 최종 벤치마크가 아님을 README·발표 자료에 명시 |
| **코드 오류 가능성** | 팀이 모든 모듈을 직접 실행하고, `pytest`로 단위 테스트 통과 여부 확인 |
| **보안 취약점** | API 키·토큰을 AI 프롬프트에 노출하지 않음. `.env`는 `.gitignore` 처리 |
| **편향 가능성** | 합성 데이터로 구성된 파일럿 실험이므로 실제 도메인 데이터와 분포 차이 존재할 수 있음 — 향후 자연 conflict 데이터(WikiContradict 등)로 검증 예정 |

## 6. 책임성·윤리성

- **AI 생성 결과 무비판 수용 금지**: 모든 코드·요약·수치는 팀이 직접 실행하거나 원문과 대조하여 검증
- **데이터 보안**: 개인정보·API 키·비공개 데이터를 AI 도구에 입력하지 않음. `.env` 파일은 `.gitignore`로 추적 제외
- **저작권·라이선스**: AI 생성 코드에 외부 라이브러리 포함 시 라이선스 확인 후 사용
- **한계 명시**: 파일럿 실험 수치는 API 모델 기반이며 최종 벤치마크가 아님을 README와 발표 자료에 명시
- **투명성 유지**: AI가 관여한 모든 작업을 본 문서에 기록. 커밋 로그에서 AI 기여 추적 가능 (`Co-Authored-By: Claude Sonnet 4.6`)
